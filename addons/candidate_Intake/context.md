# Technical Context: Candidate Intake Pipeline

This document captures the architecture, module design, and technical details of the **Candidate Intake Pipeline** module for Odoo 17.

## Repository Overview & File Structure

The module lives in the `candidate_Intake` directory and contains the following structure:

```
candidate_Intake/
├── __init__.py               # Python initialization (imports models and controllers)
├── __manifest__.py           # Odoo module manifest (metadata, dependencies, view registration)
├── controllers/
│   ├── __init__.py           # Controller package initialization
│   └── main.py               # HTTP routes for website form submission
├── models/
│   ├── __init__.py           # Models package initialization
│   └── candidate_intake.py   # Primary backend model (hr.candidate.intake) and business logic
├── security/
│   ├── ir.model.access.csv   # Access control lists (ACL) for Reviewer and Manager groups
│   └── security.xml          # Group definitions and global record rules
└── views/
    ├── candidate_intake_views.xml  # Backend tree and form views, including Chatter and buttons
    └── website_form.xml      # Website apply form template and success page
```

---

## 1. Module Manifest (`__manifest__.py`)
- **Depends on:** `['base', 'hr', 'hr_recruitment', 'website', 'mail']`.
- Registers security and view XML files under the `data` list.
- Installs as an extension (`'application': False`) for the recruitment app.

---

## 2. Data Model (`hr.candidate.intake`)
Inherits:
- `mail.thread` & `mail.activity.mixin` (provides Chatter and Activity management).

### Fields
- **Candidate Metadata:**
  - `name`: Char (tracked)
  - `email`: Char (tracked)
  - `phone`: Char
  - `linkedin_url`: Char (LinkedIn Profile URL)
  - `degree`: Char (Academic degree name)
- **Status & Source:**
  - `source`: Selection (`manual`, `referral`, `other`)
  - `status`: Selection (`new`, `review`, `approved`, `rejected`), default=`new` (tracked)
- **Recruitment Integration:**
  - `job_id`: Many2one (`hr.job`, target position)
  - `applicant_id`: Many2one (`hr.applicant`, linked recruitment card, readonly)
- **Uploaded Document:**
  - `resume_file`: Binary (stores base64-encoded PDF resume)
  - `resume_filename`: Char (stores the original file name)
- **AI & Text Data:**
  - `resume_text`: Text (parsed plain text of the resume)
  - `ai_summary`: Text (summary generated by Groq)
  - `ai_score`: Integer (AI match rating 0-100)
  - `notes`: Text (reviewer comments)

---

## 3. Controllers & Routes (`controllers/main.py`)
- **`/jobs/apply` (GET):** Renders the candidate application page on the website. Populates the "Position" dropdown with active job positions search results.
- **`/jobs/submit` (POST):**
  - Captures form text inputs and the uploaded `resume_file` multipart file.
  - Base64 encodes the file bytes and stores the filename.
  - Creates the `hr.candidate.intake` record using `sudo()` to bypass security for public users.

---

## 4. Workflows & Business Logic

### A. PDF Text Extraction & AI Analysis (`action_analyze_resume`)
1. Decodes `resume_file` from base64 back into bytes.
2. Reads the stream using `PyPDF2.PdfReader` to extract plain text across all pages.
3. Overwrites `resume_text` with the extracted text.
4. Reads the `candidate_intake.groq_api_key` System Parameter.
5. Posts a structured payload requesting a JSON response containing `{"summary": "...", "score": 85}` to Groq's Chat Completion API.
6. Writes the parsed summary and score back to the record.

### B. Conversion to Applicant (`action_approve`)
1. Looks up or creates a corresponding `hr.recruitment.degree` using the candidate's custom `degree` text.
2. Creates an `hr.applicant` record, passing:
   - Candidate contact info (`name`, `email_from`, `partner_phone`).
   - Standard Odoo recruitment fields (`linkedin_profile`, `type_id` mapping to the degree).
   - AI evaluations in the description block.
3. Automatically duplicates the candidate's `resume_file` and registers it as an `ir.attachment` record targeting the newly created `hr.applicant` record.
4. Sets the candidate record status to `approved`.

---

## 5. Security & Access Control
- **Reviewers (`group_intake_reviewer`):** Read, Write, and Create privileges on candidate intake records (`1, 1, 1, 0`).
- **Managers (`group_intake_manager`):** Full privileges including deletion (`1, 1, 1, 1`) and the exclusive right to click "Approve & Convert" to convert candidates.
